# data-heap · consumer API для агентов

Канон для фоновых агентов (hermes, openclaw и др.), которые ходят в локальный
HTTP API сервиса data-heap за новым текстом. Шим для Claude — в
`.claude/skills/data-heap/SKILL.md`, для Codex CLI — в
`.agents/skills/data-heap/SKILL.md`; оба редиректят сюда. Полный how-to,
эндпоинты, формат item и сценарии — здесь.

## Когда срабатывает

Открой этот скилл, если задача звучит как одно из:

- «Забери новое из data-heap для меня» (агенту назначены `agent_slug` и `space`).
- «Обработай свежие расходы / мысли / ссылки / inbox»  — конкретный `space`
  упомянут или подразумевается ролью агента.
- «Отметь item N обработанным» / «mark processed» — обработка одного элемента
  завершена и нужно скрыть его от следующих выборок.
- Любая работа с эндпоинтами `/v1/items` и `/v1/items/processed`.

Если задача — **загнать новый текст** в data-heap (добавить заметку, отправить
файл), это не сюда: ingestion идёт только через Telegram-ботов, отдельной
ручки на запись текста в HTTP API нет. Если задача — поменять статус item для
**другого** агента, это тоже не сюда: per-agent processed-флаги независимы.

## Что нужно знать заранее

Перед действием убедись, что:

- **`agent_slug`** — стабильный идентификатор фонового агента (например,
  `hermes`, `openclaw`). Не путать со `slug` Telegram-бота-источника (тот
  живёт на стороне ingestion и в API не фигурирует). Если оператор не выдал
  `agent_slug` — остановись и спроси, не придумывай.
- **`space`** — пространство, из которого агент читает (`expenses`,
  `thoughts`, `links`, `inbox` или произвольный slug, заданный в
  `config.toml` сервиса). Один агент может ходить в несколько `space`, но
  каждый запрос фильтрует ровно один.
- **Базовый URL** — `127.0.0.1:8080` по умолчанию (см. `[daemon].http_addr` в
  `config.example.toml`). Если оператор изменил bind, забери актуальный
  адрес у него или из его env (рекомендация: `$DATA_HEAP_URL`); жёстко не
  вшивай.
- **Auth.** Нет: API рассчитан на localhost, заголовков авторизации и токенов
  не требует. Если внезапно отвечает `401`/`403` — это не data-heap, а
  что-то другое слушает порт; остановись и сообщи оператору.

## Эндпоинты

| Метод и путь | Зачем |
|---|---|
| `GET /v1/items?agent_slug=…&space=…&limit=…` | Забрать item в `space`, которые этот `agent_slug` ещё не отметил processed. Отсортированы по `id ASC` (сначала старые). |
| `POST /v1/items/processed` тело `{"agent_slug": "...", "item_id": N}` | Перевести флаг processed для пары `(agent_slug, item_id)` в `true`. Идемпотентно; на другие `agent_slug` не влияет. |

Удалённые (soft-delete) item не отдаются: если пользователь снёс заметку
через Telegram-бот, она исчезнет из выборки даже без `mark processed`.

### `GET /v1/items` — параметры

- `agent_slug` — обязательный, непустой после `trim`.
- `space` — обязательный, непустой после `trim`.
- `limit` — опциональный, по умолчанию `50`, клампится в `[1, 500]`. Значения
  вне диапазона не отвергаются, а клампятся; не полагайся на это, шли
  валидный `limit` сам.

Пустой `agent_slug` или `space` → `400 Bad Request` с JSON `{"error": "..."}`.
Отсутствующий обязательный параметр → `400` от axum-валидации.

### Формат item

Ответ — JSON-массив объектов `Item` (может быть пустым):

```json
{
  "id": 42,
  "source": "expenses-bot",
  "space": "expenses",
  "kind": "text",
  "text": "кофе 250",
  "telegram": {
    "chat_id": 100,
    "message_id": 17,
    "user_id": 555,
    "username": "alice",
    "date": 1700000000
  },
  "created_at": 1700000123
}
```

- `id` — целое, стабильный идентификатор для `POST /v1/items/processed`.
- `source` — `slug` Telegram-бота из `[[sources]]` в `config.toml` (для логов
  и трассировки; для роутинга используй `space`, не `source`).
- `space` — то же значение, что прислал в фильтре.
- `kind` — один из `text`, `link`, `caption`, `voice`. Расшифровка:
  - `text` — обычный текст сообщения.
  - `link` — текст-ссылка (бот её не выкачивает, хранит как строку).
  - `caption` — подпись под фото/видео/документом; сам файл не сохраняется.
  - `voice` — расшифровка голосового (STT уже сделан на стороне сервиса).
- `text` — всегда непустая строка; обрабатывай как обычный пользовательский
  ввод.
- `telegram` — provenance Telegram-сообщения; `user_id` и `username` могут
  отсутствовать (anonymous-чаты, каналы).
- `created_at` — Unix-секунды, момент записи в SQLite (не Telegram-дату — она
  в `telegram.date`).

### `POST /v1/items/processed`

Тело — JSON `{"agent_slug": "...", "item_id": <int>}`. Ответ — `204 No
Content` без тела. Идемпотентно: повторный mark того же `(agent_slug,
item_id)` тоже `204`. Mark одного агента не влияет на других; для каждого
`agent_slug` свой набор «обработанных».

Пустой `agent_slug` → `400`. Несуществующий `item_id` сервис принимает
молча — расценивай как «уже нечего обрабатывать», не как ошибку.

## Готовые curl-вызовы

`curl` есть у агента по умолчанию. Базовый URL держи в env, чтобы не было
магических констант:

```bash
DATA_HEAP_URL="${DATA_HEAP_URL:-http://127.0.0.1:8080}"
AGENT_SLUG="hermes"
SPACE="expenses"

# 1. Забрать пачку новых item (до 50 по умолчанию, тут явно 100).
curl -sS --fail-with-body \
  "$DATA_HEAP_URL/v1/items?agent_slug=$AGENT_SLUG&space=$SPACE&limit=100"

# 2. Отметить item обработанным после своей обработки.
curl -sS --fail-with-body \
  -X POST "$DATA_HEAP_URL/v1/items/processed" \
  -H 'content-type: application/json' \
  -d "{\"agent_slug\":\"$AGENT_SLUG\",\"item_id\":42}"
```

Флаги: `-sS` глушит progress-bar, но оставляет ошибки; `--fail-with-body`
делает `curl` exit-code ненулевым на `4xx`/`5xx`, но при этом печатает тело
ответа (где `{"error": "..."}` от сервиса) — полезно в скриптах. Если
`curl` без `--fail-with-body` (старее 7.76) — используй `--fail` и
обрабатывай тело ответа отдельно.

Если в `space` или `agent_slug` теоретически могут попасть символы,
требующие URL-encoding (на практике это slug'и из `[a-z0-9_-]`, но если
оператор задал что-то экзотическое) — используй `curl -G --data-urlencode`:

```bash
curl -sS --fail-with-body -G "$DATA_HEAP_URL/v1/items" \
  --data-urlencode "agent_slug=$AGENT_SLUG" \
  --data-urlencode "space=$SPACE" \
  --data-urlencode "limit=100"
```

## Типовой цикл агента

1. `GET /v1/items?agent_slug=…&space=…&limit=…` — забрать пачку.
2. Если массив пустой — спать до следующего тика, ничего больше делать не
   нужно.
3. Для каждого item обработать на своей стороне (запись в свою БД,
   уведомление и т.п.). Свою бизнес-логику data-heap не знает и не хранит.
4. После успешной обработки одного item — `POST /v1/items/processed` с его
   `id`. Только после того, как **запись** в свою БД зафиксирована: иначе
   при падении между шагами потеряешь item.
5. На `4xx`/`5xx` от data-heap не помечай item processed — повтори на
   следующем тике. На ошибку **своей** обработки — тем более не помечай.
6. Закончив пачку, на следующем тике повтори с тем же `agent_slug` —
   сервис вернёт следующую партию, отсортированную по `id ASC`.

Не пытайся читать массивами по тысячам — `limit` клампится в `500`. Для
больших объёмов проще шагать страницами через повторные `GET` после mark'ов:
обработал и пометил пачку — на следующем `GET` придёт следующая. Курсора и
пагинации в API нет; «следующая страница» = «то, что ещё не processed».

## Ошибки и их обработка

| Статус | Когда | Что делать |
|---|---|---|
| `200` с пустым массивом | Нет новых item для `(agent_slug, space)` | Нормально, спать до следующего тика. |
| `400` | Пустой/отсутствующий `agent_slug` или `space`; невалидный JSON в `POST` | Это баг агента, не сервиса. Прочитай `body.error`, чини вызов. |
| `500` | Сбой Storage (SQLite locked, диск, миграция) | Не помечай item processed. Повтори вызов на следующем тике. Если повторяется — эскалируй оператору, логи смотри в выводе демона. |
| `Connection refused` / нет ответа | Демон не запущен или слушает другой адрес | Проверь, что демон поднят (`data-heap config.toml`) и `DATA_HEAP_URL` совпадает с `[daemon].http_addr` из конфига. |
| `401` / `403` | Не data-heap слушает порт | Остановись и сообщи оператору: data-heap не требует auth, кто-то другой подмешался. |

`POST /v1/items/processed` для несуществующего `item_id` возвращает `204`,
а не `404` — это сознательное решение (mark идемпотентен и не зависит от
существования item). Не строй на этом проверку «существует ли item».

## Безопасность операций

- API локальный: не пробрасывай порт наружу и не ходи в него из не-localhost.
  Если нужно подключаться с другой машины — это уже не data-heap MVP,
  эскалируй оператору.
- `mark processed` необратим из агента: ручки «unmark» нет. Если случайно
  пометил не то, оператор чинит руками в SQLite. Поэтому не помечай item,
  пока не зафиксировал результат обработки на своей стороне.
- Текст item — это пользовательский ввод. Никогда не интерполируй его в
  shell, SQL или другие интерпретаторы без экранирования; обращайся с ним
  как с произвольной untrusted строкой.

## Чеклист перед действием

- [ ] Знаю свой `agent_slug` и `space` (или явный список space'ов).
- [ ] `DATA_HEAP_URL` (или эквивалент) выставлен и указывает на актуальный
      `[daemon].http_addr`; не вшил `127.0.0.1:8080` константой.
- [ ] `curl` под рукой; если в окружении его нет — эскалируй оператору, не
      изобретай альтернативу без согласования.
- [ ] Помечаю `processed` только **после** успешной фиксации обработки на
      своей стороне.
- [ ] На `4xx`/`5xx` не помечаю — повторяю на следующем тике; на повторных
      `5xx` эскалирую оператору.
- [ ] Не использую `source` для роутинга бизнес-логики — для этого есть
      `space`; `source` оставляю в логах/трассировке.
