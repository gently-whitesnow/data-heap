# data-heap

Текстовый ingestion-сервис: «свалить в кучу» мысли, финансы, ссылки и голосовые из Telegram —
сервис принимает и складывает текст, а фоновые агенты потом разбирают эту кучу и делают полезную
работу (дайджесты, разбор расходов и т.п.).

> Статус: проектирование. Это README будущего проекта — описывает целевую картину, не текущую
> реализацию.

## Зачем

Нужна простая «свалка» для входящих заметок без ритуала ведения структуры. Пользователь шлёт что
угодно в Telegram-бота, сервис сохраняет только **текст**. Дальше независимые фоновые агенты
(hermes, openclaw и др.) периодически забирают новое и обрабатывают.

Сервис намеренно глупый: его задача — **сбор и отдача текста**. Он ничего не анализирует и не хранит
результаты анализа — где хранить вывод, решает сам агент (у него своя БД).

## Как это работает

```
 Telegram-боты ──polling──▶  data-heap (демон)  ──HTTP API──▶  агенты (hermes, openclaw, …)
   (N штук)                   ├─ domain core                    каждый со своим slug + space
                              ├─ SQLite (текст)
                              └─ транскрибация голоса
```

- **Источники.** Неограниченное число Telegram-ботов, апдейты через long-polling (`getUpdates`) —
  публичный URL и webhook не нужны. Каждый бот регистрируется как `source` и жёстко привязан к
  одному пространству (`space`): `expenses`, `thoughts`, `links`, `inbox` или произвольный slug.
- **Что сохраняем.** Только текст: текст сообщений, ссылки (как текст, без выкачивания), captions,
  расшифровки голосовых, metadata Telegram-сообщения. Всё — в одной SQLite.
- **Медиа.** Фото/видео/документы/бинарники как файлы не сохраняются: есть caption — сохраняем
  caption, иначе бот отвечает, что тип не поддержан.
- **Голос.** Временно скачивается только для транскрибации и удаляется сразу после (успех или
  ошибка). Провайдер и токен — в конфиге; при ошибке пользователь узнаёт об этом в чате.
- **Потребление.** Агенты ходят в HTTP API (localhost): забрать новые item (фильтр по
  `agent_slug` + `space`) и отметить обработанными. Состояние — бинарный `processed` на пару
  `(agent_slug, item)`; агенты независимы, отметка одного не влияет на других.

## Архитектура

Clean architecture, ports & adapters. Доменное ядро (item, source, space, состояние обработки,
правила ingestion) не зависит от инфраструктуры и общается с ней только через trait-порты.
Инфраструктура — подменяемые адаптеры за этими портами:

| Порт                 | Первый адаптер           | Подмена на будущее                     |
| -------------------- | ------------------------ | -------------------------------------- |
| `Storage`            | SQLite                   | Postgres и др. — без правок ядра        |
| `IngestionSource`    | Telegram (long-polling)  | другой мессенджер / webhook / импортер  |
| `Transcription`      | mistral / openai / local_whisper / none | другой STT-провайдер     |

Telegram и SQLite в MVP — конкретные реализации, а не точки vendor lock. Порты вводим там, где
подмена реалистична (storage, source, transcription), без оверинжиниринга на каждый слой.

## Конфигурация

Источники и провайдеры задаются в TOML: на каждый бот — `bot token`, `space-slug`, провайдер и токен
транскрибации. (Точный формат уточняется на этапе реализации.)

## Скиллы для агентов

В репозитории лежат скиллы (claude/codex) с инструкциями, как пользоваться HTTP API через `curl`,
по аналогии с подходом intent-скилла: получить новые item по `slug`+`space` и отметить их
обработанными.
## Технологии

Rust, единый бинарь/crate (демон: Telegram polling + HTTP API), SQLite.
