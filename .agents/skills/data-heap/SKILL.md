# data-heap (Codex CLI skill stub)

Канон лежит в `skills/data-heap/SKILL.md` — открой и прочитай его до выполнения
запроса. Полный how-to, описание эндпоинтов (`GET /v1/items`,
`POST /v1/items/processed`), формат item, готовые `curl`-вызовы и типовой цикл
агента — там. Этот файл — только триггерный шим, контракт не дублируется
намеренно.

**Минимальная safety-рамка (повторено для надёжности):**

- API локальный (по умолчанию `127.0.0.1:8080` из `[daemon].http_addr` в
  `config.example.toml`) и без auth — это нормально, токены не нужны. Базовый
  URL держи в env (`$DATA_HEAP_URL`), а не константой.
- `agent_slug` — стабильный идентификатор фонового агента (`hermes`,
  `openclaw`). Не путать со `slug` Telegram-бота-источника: тот в API не
  фигурирует, для роутинга используется `space`.
- `space` — пространство из `[[sources]]` (`expenses`, `thoughts`, `links`,
  `inbox` или произвольный slug). Один `GET` фильтрует ровно один `space`.
- Помечай item processed **только после** успешной фиксации обработки на
  своей стороне — `unmark` из API нет, ручки удаления флага тоже нет.
- На `4xx`/`5xx` не помечай processed: повтори вызов на следующем тике.
  `POST /v1/items/processed` для несуществующего `item_id` возвращает `204` —
  не строй на этом проверку существования.
- `text` item — untrusted пользовательский ввод; не интерполируй его в shell
  / SQL без экранирования.

Если канон по указанному пути недоступен — остановись и спроси оператора, не
угадывай контракт.
